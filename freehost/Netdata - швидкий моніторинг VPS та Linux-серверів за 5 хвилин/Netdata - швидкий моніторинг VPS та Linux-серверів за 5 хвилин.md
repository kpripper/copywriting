# **Netdata: швидкий моніторинг VPS та Linux-серверів за 5 хвилин**

Коли сайт працює на VPS або окремому Linux-сервері, адміністратору потрібна не тільки команда `top`. Важливо бачити використання процесора, завантаження оперативної пам'яті, дисковий простір, мережевий трафік, відповіді PHP-FPM і поведінку бази даних. Без цього діагностика несправностей часто починається вже після того, як користувачі помітили повільну роботу сайту.

Netdata підходить, коли потрібно швидко отримати моніторинг сервера без складної інфраструктури. Це не заміна observability-платформі для великого кластера, але для одного VPS, тестового стенда або невеликого production-сервера рішення зручне. Якщо ви тільки плануєте розміщення проєкту, можна почати з [VPS-хостингу Freehost](https://freehost.com.ua/ukr/vps-hosting/) і одразу підготувати базовий моніторинг VPS.

## **Навіщо моніторити сервер**

Моніторинг Linux-сервера потрібен не для красивих графіків, а для швидких відповідей на практичні питання:

* чому сайт став повільнішим;
* чи вистачає RAM після оновлення застосунку;
* чи не впирається база даних у диск;
* чи є піки мережевого трафіку;
* чи не росте навантаження на систему після cron-завдання;
* чи встигає PHP-FPM обробляти запити.

Без системи моніторингу сервера адміністратор бачить тільки поточний стан. Тут важливий контекст: що було хвилину тому, коли почався пік, який процес споживає ресурс і чи є зв'язок між CPU, диском та мережею. Для моніторингу продуктивності сервера це критично, бо проблема рідко живе в одному місці.

## **Що таке Netdata**

Netdata — це система моніторингу сервера з веб-інтерфейсом моніторингу, яка збирає серверні метрики майже в реальному часі. Після встановлення агент автоматично показує системні показники: CPU, RAM, swap, диски, мережу, процеси, служби systemd, файлові дескриптори та сокети.

Головна особливість — швидкий старт. Для базового сценарію достатньо встановити агент і відкрити порт 19999 локально або через SSH-тунель. Далі можна додавати колектори для застосунків: Nginx, PHP-FPM, MySQL, MariaDB, Redis, PostgreSQL, Docker та інших сервісів. Частина системних графіків з'являється без ручного налаштування.

## **Встановлення на Ubuntu 24.04 / Debian 13**

Для встановлення Netdata на Ubuntu 24.04 або Netdata Debian 13 найпростіше використати офіційний `kickstart.sh`. Скрипт визначає систему і на підтримуваних дистрибутивах встановлює native packages з репозиторію розробника. Для статті, тестового VPS або невеликого сервера краще явно обрати stable-канал.

<pre>
wget -O /tmp/netdata-kickstart.sh https://get.netdata.cloud/kickstart.sh
sudo sh /tmp/netdata-kickstart.sh --release-channel stable --non-interactive
</pre>

Після інсталяції перевірте сервіс:

<pre>
systemctl status netdata
netdata -v
</pre>

На тестовому сервері з Ubuntu 24.04 було встановлено версію v2.10.3. Для безпеки зручно залишити веб-панель доступною тільки локально і відкривати її через SSH-тунель:

<pre>
ssh -L 19999:127.0.0.1:19999 user@server
</pre>

Після цього у браузері відкрийте:

<pre>
http://localhost:19999
</pre>

### **Що встановити для практичного тесту**

Щоб у статті були не порожні графіки продуктивності, варто додати невеликий набір сервісів:

<pre>
sudo apt install -y nginx php-fpm php-cli php-mysql php-curl mariadb-server
sudo apt install -y stress-ng sysbench fio iperf3 apache2-utils
</pre>

Так можна показати моніторинг ресурсів сервера, контроль навантаження сервера, моніторинг PHP-FPM і моніторинг MySQL/MariaDB на реальних піках.

## **Перший запуск**

Після запуску агент відкриває локальну панель керування моніторингом. У верхній частині видно вузол, часовий діапазон, статус Live/Playing, сповіщення та швидку навігацію. У правому меню розділи згруповані за логікою: System, Applications, Network, Storage, Processes.

Перший тест можна зробити без додаткової конфігурації:

<pre>
stress-ng --cpu 2 --vm 1 --vm-bytes 256M --timeout 90s --metrics-brief
</pre>

Через кілька секунд графіки CPU, load average і RAM покажуть зміну навантаження. Саме тут добре видно сильну сторону інструмента: моніторинг у реальному часі не потрібно довго налаштовувати.

**[МІСЦЕ ДЛЯ СКРІНШОТА 1: Overview / CPU / RAM під час stress-ng]**

*Підпис до зображення:* Панель показує навантаження CPU та RAM у реальному часі: після запуску тесту `stress-ng` графіки одразу фіксують стрибок використання процесора, системного навантаження та pressure-метрик.

## **Огляд інтерфейсу**

Інтерфейс побудований навколо графіків. Кожен блок можна розгорнути, порівняти з іншим періодом, переглянути окремі dimensions або перейти до детальнішого контексту. Для адміністратора це зручно: не потрібно вручну збирати дані з `vmstat`, `iostat`, `free`, `ss` і журналів.

У базовому огляді варто звернути увагу на:

* CPU usage і system load;
* використання процесора окремими процесами;
* завантаження оперативної пам'яті та swap;
* дисковий простір і операції читання/запису;
* мережевий трафік;
* PSI-метрики, які показують тиск на CPU, пам'ять і I/O.

Для початківців це плюс: важливі системні показники не заховані в різних вкладках. Для досвідченіших адміністраторів корисні деталізація, API та можливість швидко зіставити метрики.

## **Моніторинг CPU, RAM, дисків та мережі**

Для CPU важливо дивитися не тільки на відсоток завантаження, а й на load average, steal time, iowait і pressure. Якщо CPU зайнятий на 100%, але iowait низький, причина одна. Якщо росте iowait і дискові затримки, проблема може бути в накопичувачі або базі даних. Саме так моніторинг Linux-сервера переходить від загального графіка до конкретної гіпотези.

Для дисків корисний тест `fio`:

<pre>
fio --name=netdata-disk-demo --filename=/tmp/netdata-fio-demo --size=512M --rw=randrw --bs=4k --runtime=90 --time_based --group_reporting
rm -f /tmp/netdata-fio-demo
</pre>

Після цього в розділі System → Storage → Disks видно IOPS, read/write bandwidth, utilization і затримки дискових операцій. Це практичний приклад, як моніторинг ресурсів сервера допомагає помітити вузьке місце.

**[МІСЦЕ ДЛЯ СКРІНШОТА 2: Storage → Disks під час fio]**

*Підпис до зображення:* Під час тесту `fio` на графіках видно зростання I/O pressure, затримки дискових операцій та активність накопичувача. Такий екран допомагає швидко оцінити, чи диск не стає вузьким місцем VPS під час інтенсивного читання й запису.

Для мережі варто дивитися inbound/outbound traffic, packets, dropped packets і TCP-сокети. Якщо одночасно ростуть web-запити, PHP-FPM connections і мережевий трафік, легше зрозуміти, що навантаження йде від реального HTTP-трафіку, а не від фонового процесу.

## **Контроль навантаження PHP-FPM**

Для моніторингу PHP-FPM треба увімкнути status page. У pool-конфігурації PHP-FPM має бути параметр:

<pre>
pm.status_path = /status
</pre>

Далі цей endpoint можна віддати через Nginx тільки для localhost і додати job у конфігурацію агента:

<pre>
jobs:
  - name: local
    url: http://127.0.0.1/status
</pre>

Після перезапуску з'являються графіки active connections, requests, request duration, slow requests, request CPU і request memory. Для тесту можна створити HTTP-навантаження:

<pre>
ab -n 5000 -c 40 http://127.0.0.1/
</pre>

**[МІСЦЕ ДЛЯ СКРІНШОТА 3: Applications → PHP FPM під час ab]**

*Підпис до зображення:* Графіки PHP-FPM показують тестове веб-навантаження: видно активні підключення, кількість запитів та короткі піки після запуску `ab`. Це допомагає швидко оцінити, чи PHP-FPM справляється з обробкою запитів до сайту.

Цей розділ особливо корисний для сайтів на WordPress, Laravel, Symfony або інших PHP-проєктів. Якщо активні підключення ростуть, а запити обробляються повільно, треба перевіряти `pm.max_children`, код застосунку, кешування і запити до бази даних.

## **Моніторинг MySQL/MariaDB**

Моніторинг MySQL і моніторинг MariaDB у Netdata працює через окремий collector. Для production краще створити окремого користувача з мінімальними правами:

<pre>
CREATE USER 'netdata'@'localhost';
GRANT USAGE, SLAVE MONITOR, PROCESS ON *.* TO 'netdata'@'localhost';
FLUSH PRIVILEGES;
</pre>

Для старіших версій MySQL або MariaDB може знадобитися `REPLICATION CLIENT` замість `SLAVE MONITOR`. Після цього в `go.d/mysql.conf` додають підключення до локальної бази:

<pre>
jobs:
  - name: local
    dsn: netdata@tcp(127.0.0.1:3306)/
</pre>

Для тесту зручно використати `sysbench`. Він створює таблиці, генерує запити і дає достатньо даних для графіків:

<pre>
sudo mariadb -e "CREATE DATABASE IF NOT EXISTS sbtest;"
sudo sysbench oltp_read_write --db-driver=mysql --mysql-socket=/run/mysqld/mysqld.sock --mysql-user=root --mysql-db=sbtest --tables=2 --table-size=20000 prepare
sudo sysbench oltp_read_write --db-driver=mysql --mysql-socket=/run/mysqld/mysqld.sock --mysql-user=root --mysql-db=sbtest --tables=2 --table-size=20000 --time=90 --threads=4 run
sudo sysbench oltp_read_write --db-driver=mysql --mysql-socket=/run/mysqld/mysqld.sock --mysql-user=root --mysql-db=sbtest --tables=2 cleanup
</pre>

**[МІСЦЕ ДЛЯ СКРІНШОТА 4: Applications → MySQL під час sysbench]**

*Підпис до зображення:* Netdata відстежує роботу MariaDB у реальному часі: під час тесту `sysbench` видно активні підключення, кількість SQL-запитів, мережевий обмін і службові метрики MySQL. Такий екран допомагає швидко оцінити навантаження на базу даних та помітити потенційні проблеми з продуктивністю.

У реальному проєкті ці графіки допомагають знайти не тільки перевантаження бази, а й непрямий вплив: повільний диск, нестачу RAM для buffer pool або різке збільшення select/update-запитів після релізу.

## **Сповіщення про проблеми**

Платформа має вбудований health monitoring. У документації описані pre-configured alerts для типових ситуацій: проблеми з диском, перевищення latency, нестача місця, аномальна активність, недоступність сервісів. В інтерфейсі їх видно у вкладці Alerts, а для повідомлень можна використовувати Cloud або локальні Agent notifications.

Для невеликого VPS на старті достатньо:

* перевірити вкладку Alerts;
* залишити активними стандартні правила;
* налаштувати канал повідомлень для критичних подій;
* не вимикати алерти без розуміння причини;
* регулярно дивитися, які alerts повторюються.

Автоматичні алерти не замінюють аналіз, але прискорюють виявлення перевантажень. Якщо система попереджає про диск, RAM або затримки, адміністратор реагує до того, як продуктивність веб-сайту стане проблемою для користувачів.

## **Переваги та недоліки**

### **Переваги**

Інструмент добре підходить для швидкого старту:

* встановлення Netdata займає кілька хвилин;
* більшість системних графіків з'являється автоматично;
* інтерфейс зручний для моніторингу у реальному часі;
* є колектори для PHP-FPM, MySQL, MariaDB та інших сервісів;
* можна працювати локально без публічного доступу до панелі;
* деталізація підходить і для початківців, і для адміністраторів.

### **Недоліки**

Є й обмеження:

* для застосунків потрібне окреме налаштування collector jobs;
* велика кількість графіків може спершу збивати з пантелику;
* для централізованих повідомлень і командної роботи зручніше підключати Netdata Cloud;
* на слабких VPS треба стежити за дисковим місцем і retention метрик;
* публічний доступ до порту 19999 не варто відкривати без захисту.

## **Висновок**

Netdata — практичний варіант, коли потрібен швидкий моніторинг сервера без довгого впровадження. За кілька хвилин можна отримати моніторинг VPS, побачити CPU, RAM, диски, мережу, PHP-FPM і MariaDB, а потім поступово додати сповіщення та власні колектори.

Для початківців цінність у простоті: встановив агент, відкрив веб-інтерфейс, побачив графіки продуктивності. Для досвідчених адміністраторів користь у деталях: PSI, IOPS, InnoDB, активні connections, системні показники і швидка кореляція між подіями. Саме тому Netdata варто розглядати як базовий інструмент для контролю стану сервера одразу після запуску VPS або перенесення сайту.
